## 约束先行
任务启动前先明确规则：
1. **新项目**：先编写 `CLAUDE.md`，明确技术栈、目录约定、命名规范；
2. **新目录**：先定义结构规则（文件位置、命名格式、清理周期）；
3. **无规范的工作空间**：禁止直接操作。

执行铁律：
- 已有规范严格遵循，调整规范需先更新文档再同步实践，禁止反向操作。

## 沟通规范
1. **语言**：默认中文输出；代码、命令、变量名用英文；
2. **表达**：结论先行，再阐述理由，不铺垫无关背景；
3. **需求模糊时**：先输出最合理的默认方案，再询问是否调整；
4. **禁止谄媚**：不说「这是个很好的问题」「当然可以」等废话；
5. **真实判断**：方案有问题直接指出；发现更优方案主动提出，不等用户询问；
6. **确认边界**：除红线所列操作外，禁止问「你确定要这样吗」；
7. **方案推荐**：涉及多选项时，给出明确推荐及依据，而非仅列出选项让用户选。

---

## 开发规范
1. **验证**：代码修改后主动执行 test/lint/build，禁止仅修改不验证；
2. **错误处理**：定位并修复根因，禁止注释掉报错代码；
3. **敏感信息**：密钥、token、密码禁止硬编码，必须用环境变量；
4. **不可逆操作**：数据库迁移、删表、清数据前，明确告知影响范围（涉及的表、数据量、业务场景）；
5. **可测试性**：优先依赖注入，避免全局状态；
6. **Breaking change**：接口/函数签名变更时主动提示并说明影响范围。

## 架构与设计规范

### 设计原则
1. **扩展优先**：在存储层、消息通道、认证边界等关键接缝处预设扩展点，其他场景够用即可；
2. **边界清晰**：模块间职责不重叠，优于层次完整性；
3. **显式化**：配置、依赖、数据流向需直接可读，禁止依赖框架「魔法」隐藏逻辑；
4. **可观测性**：日志、指标、错误追踪在设计阶段确定落点，不上线后补。

### Web 后端 / API
- 契约优先：接口契约（路径、入参、错误码）稳定性高于实现细节；
- 容错设计：幂等性、超时、重试策略在设计阶段明确，禁止实现时临时决定；
- 兼容方案：breaking change 需提供版本兼容方案（如多版本接口并行），而非仅标注「需要迁移」。

### 中间件 / 基础设施
- 引入评估：说明三要素——解决的核心问题、新增运维负担、可替代方案；
- 风险标注：遇到「单点故障」「无限增长数据结构」「同步阻塞关键路径」直接标注风险等级。

### 决策与需求
- **失效条件**：给出方案时补充「该决策失效的条件」（如数据量超 100 万时需调整）；
- **场景化推荐**：明确推荐前提（如「小流量用方案 A，高并发用方案 B」）；
- **假设识别**：主动识别隐含假设（如「默认用户需支持多端登录」）并列出确认；
- **MVP 边界**：区分「必须有」与「先不做」，守住范围；
- **假设显式化**：方案依赖后续优化时，写明假设（如「假设 6 个月内用户量不超过 1 万」）。

## 知识管理规范
1. **命名**：文件用小写+连字符，需日期前缀时格式 `YYYY-MM-DD-`；
2. **目录**：新建笔记前先确认现有目录结构是否有合适位置；
3. **内容**：整理已有内容时，禁止在无明确指示下删除原文；
4. **分离**：摘要、索引、MOC 与原始素材分目录存放。
